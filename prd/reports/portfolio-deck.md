# LUNA — 포트폴리오 프로젝트 발표 자료

> 비건 화장품 D2C 쇼핑몰 기획 · 개발 · 데이터 분석 · 마케팅 전략
> 프로젝트 기간: 2026.03.20 ~ 2026.03.21

## 프로젝트 개요

### 한 줄 정의

"달빛처럼, 투명하게" — 성분 투명성을 핵심 가치로 하는 비건 스킨케어 D2C 브랜드 **LUNA**의 기획부터 분석까지 전 과정을 직접 설계한 커머스 포트폴리오

### 목적

- **핵심 목적**: 커머스 데이터 분석 역량 증명 (퍼널·RFM·코호트·매출 분석)
- **부가 목적**: 기획 → 개발 → 마케팅 → 보고로 이어지는 D2C 사이클 전 단계 경험

### 맡은 역할

프로젝트 전 단계를 단독으로 수행:
브랜드 전략 수립 / 제품 기획 / PRD 작성 / 목업 데이터 생성 / 웹 앱 개발 / 분석 도구 연동 / 데이터 분석 / 마케팅 전략 / 최종 보고

### 기술 스택

| 영역 | 기술 |
|------|------|
| Frontend | Next.js 16 (App Router), React 19, Tailwind CSS, shadcn/ui |
| State | Zustand |
| Backend | Supabase (PostgreSQL, Auth, Storage) |
| Analytics | GTM, Amplitude, Mixpanel |
| Data | Python (pandas, matplotlib), SQL (PostgreSQL), Tableau |
| Deploy | Vercel |

## 프로세스 — Phase 1 ~ 8

### Phase 1: 브랜드 전략 수립

국내 비건 화장품 시장 분석(2026년 1.2조원 규모, 연 22% 성장)을 바탕으로 LUNA의 포지셔닝을 정의했다. 이니스프리·아로마티카·톤28 등 경쟁사 분석을 통해 "성분 투명성"을 핵심 차별화 포인트로 선정하고, 3종 페르소나(성분 탐구자 지수연, 미니멀리스트 박서현, 비건 입문자 이나래)를 구체화했다.

### Phase 2: 기능 기획 / PRD

홈·목록·상세·장바구니·결제·마이페이지·회원 등 7개 핵심 페이지의 기능 명세를 작성했다. GTM/Amplitude/Mixpanel 기준 16개 분석 이벤트를 정의하고, Supabase 스키마(8개 테이블)를 설계했다. KPI는 GMV·CVR·AOV·MAU·재구매율 5종으로 정의했다.

### Phase 3: 목업 데이터 생성

Python으로 현실적인 목업 데이터를 생성했다. 1,000명 유저 · 2,850건 주문(12개월 계절성 반영) · 50개 SKU · 세션 로그 · 재고 변동 · 500건 리뷰를 포함한 8개 CSV 파일을 생성하고 Supabase에 업로드했다.

### Phase 4: 웹 앱 개발

Next.js 16 App Router 기반 쇼핑몰을 구현했다. 서버/클라이언트 컴포넌트를 분리하고, Zustand로 장바구니 상태를 관리했으며, shadcn/ui로 일관된 디자인 시스템을 적용했다. 모바일 반응형으로 7개 페이지를 완성하고 Vercel에 배포했다.

### Phase 5: 분석 도구 연동

GTM 컨테이너를 웹 앱에 연동하고, 커스텀 이벤트 16종(page_view, product_view, add_to_cart, begin_checkout, purchase 등)을 구현했다. analytics 추상화 레이어를 통해 단일 호출로 GTM·Amplitude·Mixpanel에 동시 발송하는 구조를 설계했다.

### Phase 6: 데이터 분석

5종 분석(퍼널·RFM·코호트·재고·매출)을 SQL과 Python 이중으로 구현했다. RFM 세그먼테이션으로 6개 고객군을 식별하고, 코호트 리텐션 패턴에서 M0→M1 전환이 리텐션의 핵심 레버임을 확인했다. Tableau 4개 시트로 최종 대시보드를 구성했다.

### Phase 7: 마케팅 전략

Phase 6 분석 결과를 바탕으로 데이터 기반 4개 캠페인을 설계했다: 신규유망 2회차 유도 캠페인, 슬리퍼 리타겟팅 캠페인, 충성고객 VIP 전환 캠페인, 성분 교육 콘텐츠 캠페인. 채널별(인스타그램·카카오·이메일) 카피라이팅 자산과 1개월 콘텐츠 캘린더를 함께 작성했다.

### Phase 8: 최종 보고

포트폴리오 발표 자료·최종 인사이트 문서·GitHub README를 작성하며 전체 프로젝트를 마무리했다.

## 주요 분석 인사이트

### 인사이트 1 — VIP 8.4%가 매출의 핵심 (파레토 법칙)

전체 969명 구매 고객 중 단 81명(8.4%)의 VIP 고객이 평균 99만원, 13회 구매를 기록했다. 이 고객군의 이탈 방지와 충성고객(11.7%)의 VIP 전환이 매출 방어의 최우선 과제다. 추천인 프로그램과 얼리액세스 혜택으로 VIP를 록인하고, 구매 횟수 7회 도달 시 VIP 자동 승급 구조를 설계했다.

**추천 액션**: VIP 전용 멤버십 티어 도입, 번들 상품으로 충성고객 AOV 상승 유도

### 인사이트 2 — 슬리퍼 30.2% 리타겟팅으로 매출 30% 증대 가능

293명(30.2%)의 슬리퍼 고객은 평균 11만원 1회 구매 후 114일간 미구매 상태다. 전체 고객의 3분의 1이 잠든 상태이며, 이 중 20%만 2회차 구매로 전환해도 약 646만원(293명 × 20% × 11만원)의 추가 매출이 발생한다. 카카오 친구톡 재방문 캠페인과 "다시 만나요" 쿠폰이 가장 직접적인 레버다.

**추천 액션**: 마지막 구매 후 60일 기준 자동 리타겟팅 캠페인 트리거 설정

### 인사이트 3 — M0→M1 전환율이 리텐션의 핵심 레버

코호트 분석에서 M0(가입 월) 구매율은 초기 코호트 12.5%에서 최근 95.7%로 크게 개선됐으나, M1 리텐션은 16~76%로 편차가 크다. 3회 이상 구매한 고객은 장기 리텐션율 20~35%로 안정화되는 패턴을 보인다. 즉 가입 후 첫 30일 내 2회차 구매를 유도하는 것이 장기 고객 전환의 핵심이다.

**추천 액션**: 웰컴 이메일 시리즈 자동화 (D+3·D+7·D+14·D+21 시점 개인화 발송)

## 기술적 도전과 해결

### 도전 1 — Next.js 16 App Router 서버/클라이언트 컴포넌트 분리

**문제**: App Router에서 서버 컴포넌트와 클라이언트 컴포넌트의 경계 오류, `useState`/`useEffect` 혼재로 빌드 실패가 반복됐다.

**해결**: 데이터 페칭은 서버 컴포넌트에서, 인터랙션(장바구니·필터·모달)은 클라이언트 컴포넌트로 명확히 분리했다. `"use client"` 지시어를 컴포넌트 트리의 최하단에만 적용해 불필요한 클라이언트 번들을 최소화했다.

### 도전 2 — Analytics 추상화 레이어 (GTM + Amplitude + Mixpanel 동시 발송)

**문제**: 각 도구의 SDK를 페이지마다 직접 호출하면 코드 중복이 심하고, 이벤트 스키마 변경 시 전체 수정이 필요했다.

**해결**: `web/src/lib/analytics/` 디렉토리에 추상화 레이어를 구성했다. `track(eventName, properties)` 단일 함수 호출로 GTM dataLayer push → Amplitude SDK → Mixpanel SDK 순으로 동시 발송하는 구조를 설계했다. 이벤트 이름과 프로퍼티 타입은 TypeScript 인터페이스로 강제했다.

### 도전 3 — RFM 세그먼테이션 SQL + Python 이중 구현

**문제**: SQL만으로 RFM 점수를 계산하면 실시간 확인이 어렵고, Python만으로 하면 Tableau 연동 시 성능 문제가 발생한다.

**해결**: SQL(`data/queries/02_rfm_analysis.sql`)로 Supabase `rfm_segments` 요약 테이블을 생성해 Tableau와 직접 연결하고, Python(`data/scripts/rfm_analysis.py`)으로 CSV를 출력해 오프라인 분석과 시각화에 활용했다. 두 결과를 교차 검증해 정합성을 확인했다.

## 회고 & 향후 계획

### 배운 점

- **데이터 파이프라인 설계의 중요성**: 분석 쿼리를 작성하기 전에 스키마와 목업 데이터의 현실성을 충분히 검토해야 한다는 것을 체감했다. 초반의 스키마 설계 퀄리티가 분석의 깊이를 결정한다.
- **추상화의 가치**: Analytics 추상화 레이어처럼 공통 인터페이스를 먼저 설계하면 도구를 교체하거나 추가할 때 비용이 크게 줄어든다.
- **세그먼트 기반 사고**: 전체 평균보다 세그먼트별 패턴에서 훨씬 날카로운 인사이트가 나온다. "평균의 함정"을 피하는 것이 분석의 핵심이다.

### 아쉬운 점

- **실제 트래픽 데이터 부재**: 목업 데이터를 사용해 실제 유저 행동과의 괴리가 있다. GTM·Amplitude 연동 코드는 작성했으나 실제 이벤트 누적 데이터를 분석하지 못했다.
- **Tableau 대시보드 퍼블리시 미완성**: Tableau Public 퍼블리시까지 완료하지 못해 대시보드 링크를 README에 포함하지 못했다.
- **결제 플로우 모킹**: 실제 PG 연동 없이 모킹으로 처리해 실결제 데이터를 확보하지 못했다.

### 향후 계획

1. **실 데이터 누적**: Vercel 배포 후 실제 방문자 데이터(GTM·Amplitude)를 30일 이상 누적해 목업 데이터와 비교 분석
2. **Tableau Public 퍼블리시**: Tableau 대시보드를 퍼블리시하고 README에 링크 추가
3. **정기구독 기능 추가**: Supabase Edge Function + 외부 결제 SDK(토스페이먼츠)로 실제 정기결제 플로우 구현
4. **A/B 테스트 설계**: Amplitude Experiment를 활용해 랜딩 페이지 CTA 버튼 A/B 테스트 설계 및 실행
